# 专属问答 AI 助手通用训练模板

> 用途：把一个普通资料库，训练成“有身份、有边界、有证据链、可持续学习”的专属问答 AI 助手。  
> 适用对象：Claude Code、Copilot CLI、Trae、ChatGPT 类 Agent，或任何“Agent + LLM + 本地知识库”组合。  
> 核心目标：让 AI 在**每次新会话都是干净上下文**的前提下，仍能快速知道“我是谁、我能回答什么、我必须引用什么、我该去哪里找资料、找不到时该怎么说”。

---

## 0. 先理解：你要训练的不是“会背文档的 AI”，而是“会按规则查证并回答的 AI”

一个好用的专属问答助手，通常不是靠一大堆零散资料堆出来的，而是靠下面 5 个能力同时成立：

1. **身份稳定**  
   每次新会话一进来，AI 就知道自己是谁、服务哪个领域、哪些内容能答、哪些内容不能乱答。

2. **证据优先**  
   AI 不直接凭语言模型“猜”，而是优先去本地文档、源码、FAQ、索引里找依据。

3. **结构化检索**  
   AI 不是把整个知识库全文塞进上下文，而是通过初始化索引、技能路由、关键词检索、语义检索、MCP 工具按需取用。

4. **持续学习**  
   新资料来了以后，能抓取、整理、摘要、审计、索引、复用，而不是靠人工每次重新喂。

5. **回答受约束**  
   AI 不只是“能检索”，还知道什么时候证据足够、什么时候必须引用、什么时候该明确说 `未在当前知识库找到依据`。

如果这 5 点缺一，AI 往往会变成：

- 角色不稳定：今天像工程师，明天像客服，后天像搜索引擎
- 回答不一致：同一个问题每次说法不同
- 容易幻觉：没依据也说得像真的
- 难以维护：知识更新后，旧答案还在乱飞
- 检索看起来很强，但一到最终回答还是会飘

---

## 1. 通用架构总览

建议把你的专属助手拆成 6 层：

### 第 1 层：初始化层
负责告诉 AI：

- 你是谁
- 你的知识来源优先级是什么
- 什么必须引用
- 什么属于推断
- 什么超出范围
- 知识库的大地图在哪里

**这一层的目标不是“讲知识”**，而是**建立行为合同**。

---

### 第 2 层：技能 / Prompt 路由层
负责把用户问题分类，然后只加载当前问题真正需要的专业提示。

例如：

- 用户问“参数配置” → 加载参数查询 skill
- 用户问“为什么失败” → 加载排障 skill
- 用户问“这个函数在哪” → 加载代码导航 skill
- 用户问“帮我回客户邮件” → 加载客户回复 prompt

**这一层的目标不是重复知识库**，而是减少 token、提高命中率、统一输出结构。

---

### 第 3 层：结构化知识层
把 PDF、源码、社区经验、FAQ、人类经验总结成 AI 容易理解的 Markdown 文档。

这些 Markdown 的作用不是“存档”，而是变成 AI 的**二次加工知识层**：

- 比原始 PDF 更短
- 比源码更适合解释
- 比零散笔记更有逻辑
- 比单纯 FAQ 更可扩展

---

### 第 4 层：检索与工具层
负责让 AI 真正“拿到资料”。

通常包含：

- 关键词检索
- 符号检索
- 全文分块索引
- 向量语义检索
- 查询改写 / 扩展
- 显式 reranker
- grounded answer / 证据包
- MCP Server
- 命令行检索脚本

**这一层解决的是“AI 知道去哪找证据”的问题。**

---

### 第 5 层：资料摄取 / 持续学习层
负责把新资料变成知识库的一部分。

例如：

- 抓官网 FAQ / Community 帖子
- 导入新 PDF
- 生成摘要与候选 FAQ
- 更新索引
- 重建向量库
- 输出验收报告

**这一层解决的是“知识库如何继续长大，而且长得有秩序”。**

---

### 第 6 层：治理与交付层
负责保证这套系统不是一次性实验，而是可交接、可验证、可升级的“知识产品”。

通常包含：

- 使用手册
- README 门面
- 审计报告
- 交付清单
- 执行报告
- 会话状态记录
- 下一轮改进计划
- 检索评测报告
- grounded answer 约束评测

**这一层解决的是“人类训练者和后续维护者能不能接得住”。**

---

## 3. 每类关键文件到底是干什么的

这一节最重要。训练另外一个 AI 时，不能只告诉它“有这些文件”，而要告诉它“这些文件分别意味着什么”。

---

### 3.1 `CLAUDE.md`：每次新会话的“身份启动器”

这是整个系统的入口。

但要注意：**它是“约定式入口”，不是“所有 Agent 的通用强制标准”**。

- 有些工具会自动读取它
- 有些工具会读取别的约定文件
- 还有些工具不会自动读取任何仓库初始化文件，只能手动粘贴 Prompt

所以更稳妥的设计不是“假设所有工具都支持 `CLAUDE.md`”，而是：

1. 把仓库根目录初始化文件当成 **canonical source**
2. 再为不同 Agent 准备 **兼容镜像/适配入口**
3. 再额外准备 **手动 Prompt 入口**

它应该解决 5 个问题：

1. **我是谁？**
2. **我的领域是什么？**
3. **我的知识从哪里来？优先级如何？**
4. **我回答时必须遵守什么规则？**
5. **我需要先知道哪些快速索引？**

#### 这个文件应该包含什么

- 领域身份声明
- 知识来源优先级
- 回答规范
- 引用规则
- 禁止编造的边界
- 关键产品边界 / 适用范围
- 快速索引
- 可用 Skills / 工具
- 新会话初始化清单
- 对高风险回答的最小约束

#### 你不要在里面写什么

- 不要塞过多细节知识
- 不要复制整本 FAQ
- 不要把所有源码逻辑塞进去

因为它的职责是“**定规矩 + 给导航**”，不是“**承载所有知识**”。

#### 更稳妥的工程实践

建议你不要把“某个工具是否自动读取 `CLAUDE.md`”当成系统成立的前提。

更好的抽象方式是：

- `CLAUDE.md`：总初始化规范
- `.claude/CLAUDE.md`：给 Claude 生态的兼容镜像
- `AGENTS.md` / `.github/copilot-instructions.md` / 其他平台约定文件：给其他生态的适配镜像
- `prompts/main.md`：给完全不支持自动读取的工具做手动入口

也就是说，**真正通用的不是某个文件名，而是“canonical init doc + adapter layer”这个方法论**。

#### 额外经验：不要把“自动读取初始化文件”当成唯一前提

更稳妥的方式是同时准备三种入口：

1. **自动入口**：`CLAUDE.md`、`AGENTS.md`、`.github/copilot-instructions.md`
2. **手动入口**：`prompts/main.md`
3. **人类入口**：根目录 `README.md`

这样即使某个工具不自动读取初始化文件，人类维护者和使用者也能快速知道：

- 这个助手是什么
- 该怎么提问
- 索引需不需要重建
- MCP 是不是要自己配

#### 通用写法示例

```md
# [领域名] 专属 AI 助手

## 身份声明
你是 [领域名] 专属 AI 助手，负责回答 [产品/系统/协议/代码库] 相关问题。

## 知识来源优先级
1. 本地仓库文件（最高优先级）
2. 官方手册 / 原始 PDF / 源码
3. 通用工程经验（必须标注【推断/经验】）
4. 禁止编造未验证的参数、寄存器、接口定义

## 回答规则
- 关键结论必须引用：`文件路径:行号` 或 `文档名#页码`
- 推断内容必须标注 `【推断/经验】`
- 没找到依据时，必须明确说“未在当前知识库找到依据”

## 快速索引
- 架构概述：`knowledge/overview.md`
- 常见问题：`knowledge/faq.md`
- 排障手册：`knowledge/troubleshooting_playbook.md`
- 参数目录：`knowledge/parameters_catalog.md`
```

---

### 3.1.1 `README.md`：给人类看的“工作区门面”

很多专属助手工作区只有 `CLAUDE.md` 和 `knowledge/*.md`，却没有一个给人类看的总入口。  
这会导致训练者知道怎么用，但后续接手的人一进来完全不知道：

- 这是个什么工作区
- 用户是不是直接自然语言提问就行
- RAG 要不要自己启动
- MCP 要不要手动跑

推荐在根目录加一个简洁的 `README.md`，至少回答 6 件事：

1. 这是什么工作区
2. 这个专属助手擅长什么
3. 用户是不是只要自然语言提问就行
4. 需不需要手动运行 RAG / MCP
5. 维护者什么时候要重建索引
6. 详细说明文档在哪里

一个很实用的经验是：

- `README.md`：先让人上手
- `knowledge/usage_guide.md`：把细节说清楚

---

### 3.2 `.claude/skills/*/skill.md`：渐进式专业子人格

Skill 的本质不是“另一份知识库”，而是：

- **缩小问题空间**
- **缩短上下文**
- **固定回答套路**
- **强调某类问题的处理流程**

#### 一个好 Skill 应该包含什么

- 触发条件
- 角色定义
- 核心规则
- 该类问题的检索策略
- 输出模板
- 常见错误
- 边界声明

#### 举例

如果你做的是数据库专属助手：

- `db-main`：总入口，识别“性能 / SQL / 迁移 / 备份 / 故障”
- `db-troubleshoot`：专门处理“慢查询、锁等待、连接失败”
- `db-schema-navigate`：专门定位表结构、索引、迁移脚本
- `db-param-lookup`：专门查参数、默认值、版本差异

#### 通用 Skill 模板

```md
# [Skill 名称]

## 触发条件
- 当用户提到：[关键词1] [关键词2] [关键词3]

## 角色定义
你是 [这个领域下的某类专家]。
你的任务是：
1. 分类问题
2. 提取关键信息
3. 去指定资料中找证据
4. 按固定格式输出

## 核心规则
- 不在信息不足时给出确定性结论
- 所有关键结论必须引用
- 推断必须标注【推断/经验】

## 检索策略
- 先查：`knowledge/[某文档].md`
- 再查：`docs/`
- 再查：源码 / 配置文件 / 日志

## 输出格式
【结论】
【证据引用】
【操作步骤】
【风险提示】
【下一步所需信息】
```

### 3.3 `prompts/*.md`：平台无关的提示词版本

不是所有平台都会自动读取 `CLAUDE.md` 或 Skills。

所以你需要一套“可复制粘贴”的 Prompt 文件，供其他 Agent 使用。

#### 它们的作用

- 给不支持 Skill 的平台一个手动初始化入口
- 把某类任务的提示词单独封装出来
- 便于跨平台复用

#### 常见 prompt 文件及作用

| 文件 | 用途 |
|------|------|
| `prompts/main.md` | 主入口提示词，适合通用问答 |
| `prompts/troubleshooter.md` | 故障诊断专用 |
| `prompts/code_navigator.md` | 定位代码、函数、宏、调用链 |
| `prompts/customer_reply.md` | 生成发给客户/用户的正式回复 |

#### `prompts/main.md` 应该写什么

- 身份
- 证据优先级
- 问题分类规则
- 输出格式
- 无证据时怎么说

#### `prompts/customer_reply.md` 应该写什么

- 角色是技术支持工程师，而不是单纯解释器
- 语气要求
- 邮件结构
- 不能过度承诺
- 结尾必须索取下一步信息

---

### 3.4 `knowledge/overview.md`：知识库总地图

这个文档不是“概述而已”，它的真正作用是：

- 告诉 AI 这个系统由哪些部分构成
- 数据流 / 架构图是什么
- 关键文件在哪里
- 哪些模式、版本、产品边界不能混淆
- 当前知识缺口是什么

#### 这个文件应该长什么样

```md
# [项目名] 一页纸总览

## 1. 应用目标
这个系统是做什么的。

## 2. 系统框图
上游 -> 核心模块 -> 下游

## 3. 关键文件地图
| 文件/路径 | 类型 | 用途 |

## 4. 工作模式 / 运行模式
| 模式 | 配置 | 说明 |

## 5. 构建 / 运行 / 验证步骤（摘要）

## 6. 当前知识缺口
```

#### 为什么它很重要

因为 AI 即使知道很多细节，也常常不知道**这些细节在系统里是什么位置**。  
`overview.md` 负责建立全局坐标系。

---

### 3.5 `knowledge/faq.md`：把“高频问题”沉淀成标准答案

FAQ 不是随手记几个问答，而是把重复出现的问题收敛成统一口径。

#### 它应该覆盖什么

- 用户反复问的问题
- 最容易误解的概念
- 典型踩坑
- 最常见配置错误
- 最常见故障现象

#### 推荐每个 FAQ 的结构

```md
## Q1：问题标题

### 结论
一句话回答。

### 适用前提
说明在什么模式 / 版本 / 产品下成立。

### 证据
- `文档#页码`
- `文件路径:行号`

### 解释
用人能看懂的话解释原因。

### 建议动作
告诉用户下一步做什么。
```

#### 典型例子

- “为什么吞吐达不到宣传值？”
- “为什么这个标志位不按预期变化？”
- “为什么某模式下配置 A 能用、配置 B 不能用？”
- “为什么代码改了参数但运行无变化？”

---

### 3.6 `knowledge/troubleshooting_playbook.md`：排障决策树

这个文件是“故障诊断流程”，不是知识综述。

#### 它要解决的核心问题

当用户说“它不工作”时，AI 不应该乱猜，而应该：

1. 先把症状分类
2. 再追问关键上下文
3. 再按优先级排查已知高概率原因
4. 每一步都尽量有依据

#### 推荐写法

```md
# 排障手册

## 症状分类
| 类别 | 症状 |
|------|------|
| A | 无法启动 |
| B | 数据错误 |
| C | 性能不足 |
| D | 时序异常 |

## Step 1：必问信息
- 当前版本
- 当前配置
- 错误现象
- 日志 / 波形 / 报错

## Step 2：按类别排查
### A 类：启动失败
1. 检查 ...
2. 检查 ...

### B 类：数据错误
1. 检查 ...
2. 检查 ...
```

#### 你训练别的助手时也适用

比如：

- 驱动助手：蓝屏 / 装不上 / 功能异常 / 性能差
- 数据库助手：连接失败 / 慢查询 / 主从延迟 / 数据不一致
- 编译系统助手：编译失败 / 链接失败 / 运行崩溃 / CI 失败

### 3.7 `knowledge/parameters_catalog.md`：参数/配置/宏目录

这个文件是“查参数字典”。

#### 它应该包含什么

- 参数名
- 默认值
- 可选值
- 作用
- 模式依赖
- 版本差异
- 修改位置
- 修改后的风险

#### 推荐结构

```md
# 参数目录

## 时钟参数
| 参数 | 默认值 | 范围 | 说明 | 来源 |

## 缓冲参数
| 参数 | 默认值 | 影响 | 修改位置 |

## 模式开关
| 宏/配置 | 可选值 | 含义 | 文件位置 |
```

#### 为什么 AI 需要这个文件

原始源码里参数经常是分散的，AI 即便能搜到，也未必能迅速判断：

- 默认值是什么
- 和哪个模式绑定
- 改大 / 改小的副作用是什么

这个文档就是把“散落的配置定义”整理成“可解释的参数目录”。

---

### 3.8 `knowledge/api_reference.md`：常用接口说明

这个文件适合任何存在“函数 / API / 命令 / RPC / CLI 参数”的领域。

#### 适用场景

- SDK API
- Python 库函数
- CLI 命令
- REST API
- 寄存器配置函数

#### 推荐内容

- 接口名
- 参数含义
- 返回值 / 错误码
- 前置条件
- 常见误用
- 最小示例

#### 模板

```md
## ApiName / CommandName

### 用途
这个接口解决什么问题。

### 参数说明
| 参数 | 类型 | 含义 | 常见值 |

### 返回值 / 错误
| 返回 | 含义 |

### 常见坑
- ...

### 例子
<这里放一个最小可运行示例，例如命令行、代码片段或请求示例>
```

---

### 3.9 `knowledge/timing_analysis.md` 或领域深度专题文档

这个名字可以换，但职责不要变：  
它是“对某一类复杂核心问题做深度展开”的专题文档。

#### 在不同领域中，它可能是什么

| 领域 | 对应专题文档 |
|------|--------------|
| 硬件接口 | `timing_analysis.md` |
| 数据库 | `query_planning.md` |
| 编译工具链 | `build_pipeline_analysis.md` |
| 多线程系统 | `concurrency_model.md` |
| 网络协议 | `packet_flow_analysis.md` |

#### 这种文档应该写什么

- 关键公式 / 关键时序 / 关键流程
- 表格化参数
- 典型时序图 / 状态图
- 多种模式的差异
- 容易误判的地方
- 通过源码 / 原始文档验证出来的结论

---

### 3.10 `knowledge/audit_report.md`：防幻觉的关键文件

这个文件非常重要，很多知识库都没有这一层。

#### 它的作用

不是新增知识，而是**检查知识库里最关键的结论到底对不对**。

#### 为什么必须有

因为知识库里的二次整理文档本身也可能写错。  
AI 如果只相信“整理后的文档”，还是会继承人的错误。

#### 建议做法

选 10~30 条最关键的高频结论，逐条做：

1. 知识文件里怎么说
2. 原始 PDF / 代码 / 文档原文怎么说
3. 是否一致
4. 有哪些前提条件
5. 是否有歧义，需不需要修正

#### 推荐结构

```md
# 正确性审计报告

## 当前覆盖状态
- 已覆盖哪些资料
- 还缺什么资料

## 关键结论核验
### A-01 某结论
- 知识文件引用：
- 原文引用：
- 核验结论：
- 前提条件：
```

这份文档的存在，会显著降低 AI“把二手总结当绝对真理”的风险。

---

### 3.11 `knowledge/usage_guide.md`：给人类训练者和使用者看的手册

这个文件不是给 AI 本身看的，更多是给人类团队看的。

#### 它应该回答

- 我怎样提问，AI 回答最准？
- 我需要提供哪些上下文？
- AI 的固定输出结构是什么？
- 哪些问题它最擅长？
- 哪些问题要补充日志、波形、代码片段？

#### 它的价值

用户提问质量越高，AI 回答越稳定。  
所以这个文件本质上是在训练“人如何正确使用 AI”。

#### 建议和 `README.md` 分工

- `README.md`：工作区门面，适合第一次进入的人
- `usage_guide.md`：更完整的使用手册，适合已经知道这是什么的人

#### 建议额外写清楚的 3 件事

1. 用户默认是否只需要自然语言提问
2. 维护者什么时候需要重建索引
3. MCP 是由宿主自动托管，还是需要手动配置

---

### 3.12 `knowledge/improvement_plan.md`：知识库成长路线图

这个文件用来记录：

- 目前知识覆盖到哪里
- 还有哪些缺口
- 哪些改进 ROI 最高
- 下一轮该补什么资料

#### 推荐写法

按优先级列出：

- 缺什么资料
- 解决什么问题
- 产出什么文档 / 脚本 / 工具
- 验收标准是什么

这可以让你的知识库从“能用”逐步走向“成熟”。

---

### 3.13 `memory/SESSION_STATE.md`：帮助训练者理解当前系统状态

这类文件适合记录：

- 已完成什么
- 未完成什么
- 当前有哪些阻塞
- 下一步建议

它更像“项目状态面板”，不是直接给终端用户看的知识文件。

更推荐把状态拆成两层：

- `governance/CURRENT_STATUS.md`：当前真值状态
- `memory/SESSION_STATE.md`：历史会话快照 / 归档

这样可以避免模型刷新、网络中断、人工补记后，旧状态文件被误当成当前状态。

## 4. 为什么要同时保留“原始资料”和“知识库 Markdown”

很多人会问：既然都整理成 Markdown 了，为什么还保留 PDF、源码、原帖？

答案是：**因为 AI 需要二级知识，人类和系统需要一级证据。**

### 一级证据层

- 官方 PDF
- 官方 API 文档
- 原始源码
- 原始论坛帖 / KBA / 工单记录

### 二级知识层

- FAQ
- 参数目录
- 排障手册
- 专题分析
- 使用指南

### 正确关系

不是“二选一”，而是：

> 一级证据提供真实性，二级知识提供可理解性。

## 5. 检索层怎么设计，AI 才真的会“找资料”

只放文档，不做检索层，AI 很容易找不到东西。

建议至少准备 4 类检索能力：

### 5.1 符号检索
适合查：

- 函数名
- 宏定义
- 类名
- 变量名
- RTL 信号名
- 配置项名

输出通常是：

- 名称
- 文件路径
- 行号
- 类型

---

### 5.2 文本分块检索
适合查：

- PDF 某一页内容
- 源码某一段上下文
- 文档中的关键词段落

做法通常是：

- 把文档和源码切成若干 chunk
- 建立 `text_chunks.jsonl`
- 每个 chunk 带上 source / page / line 等 metadata

---

### 5.3 向量语义检索
适合查：

- 用户问法和文档用词不一致
- 用户用描述性语言而不是标准术语
- 英文/中文/近义词混合提问

例如用户问：

- “为什么写完之后 flag 还不变”
- “为什么这个接口看起来像堵住了一样”

这种问题单靠关键词常常不好命中，语义检索更有效。

---

### 5.4 查询改写 / Rerank / grounded answer

如果你只做“召回”，不做“回答约束”，AI 仍然会在最终回答时飘。

因此更成熟的 RAG，建议再补 3 类能力：

#### 查询改写 / 扩展

适合处理：

- 用户问法很口语
- 用户用“为什么 / 怎么 / 区别 / 哪里”提问
- 用户的中文表达和文档术语不一致

#### 显式 reranker

适合处理：

- 召回候选已经很多，但排序还不够稳
- 文档标题、路径、metadata 与问题高度相关，需要二次加权

#### grounded answer / 证据包

适合处理：

- 高风险技术判断
- 客户回复草稿
- 需要严格引用
- 想先判断“证据够不够回答”

这种工具的目标不是替代最终回答，而是先给上层 Agent 一份：

- 当前问题的查询计划
- 推荐证据
- 最少引用条数
- 证据不足时的兜底规则

---

### 5.5 MCP / 工具接口

如果你的 Agent 支持 MCP 或工具调用，建议把常见查询能力封成工具：

- `semantic_search`
- `get_faq`
- `get_parameters`
- `get_timing`
- `get_troubleshooting`
- `grounded_answer`

这样 AI 就不是“记住所有知识”，而是“需要时自动调用”。

---

## 6. `scripts/` 和 `ingest/` 不只是工具目录，而是“持续学习流水线”

如果你希望这个助手越用越强，就必须设计“新资料如何进入系统”。

---

### 6.1 `ingest/` 的意义

这是新资料的缓冲区。

例如：

- `ingest/urls.txt`：准备抓的网页
- `ingest/html/`：抓下来的网页 Markdown
- `ingest/pdf/`：新导入的 PDF
- `ingest/md/`：人工整理的离线资料
- `ingest/auth/`：登录态 / cookies / 抓取授权状态

它的职责是：

> “先把外部资料收进来，再决定如何整理进知识库。”

如果你的工作区后期已经形成了更稳定的权威源，也可以把 `urls.txt` 进一步收敛成：

- `urls_checked.txt`：人工审核后的权威清单
- `url_success.txt`：实际抓取成功
- `url_missing.txt`：实际抓取失败及原因
- `fetch_audit.json`：逐条状态审计

---

### 6.2 `scripts/init_kb.py`

用于第一次建库。

它一般做这些事：

- 提取源码符号
- 提取 PDF 文字
- 切分源码 / 文档 chunk
- 生成基础索引

适合首次初始化整个知识库。

---

### 6.3 `scripts/update_kb.py`

用于增量更新。

它一般负责：

- 只处理新文件
- 追加到现有索引
- 记录哪些文件处理过

适合日常维护，不需要每次全量重建。

---

### 6.4 `scripts/build_vector_index.py`

把 `text_chunks.jsonl` 变成向量索引库。

它解决的是：

- 语义检索如何落地
- 不同文档如何统一被召回

---

### 6.5 `scripts/fetch_web.py`

把网页、社区帖子、知识库文章抓成本地资料。

常见模式包括：

- public：公开抓取
- login：带登录态抓取
- auto：先公开后回退登录

---

### 6.6 `scripts/run_ingest_pipeline.py`

建议提供一个“一键流水线”脚本，把以下步骤串起来：

1. 抓取
2. 摘要
3. 索引
4. 向量化

这样新资料进来后，人和 AI 都不需要手动跑很多命令。

---

### 6.7 `scripts/verify_ingest.py`

很多项目有“导入脚本”，但没有“验收脚本”。  
这是不够的。

你最好验证：

- 新文件是否真的抓到了
- digest / FAQ 候选是否生成
- 索引里是否包含新内容
- 向量库是否存在
- 失败日志是否过多

---

## 7. 如何给另一个 AI 一份“能听懂”的初始化模板

下面这段可以作为你训练其他专属助手时的骨架：

```md
# [领域名] 专属 AI 助手初始化文件

## 你的角色
你是 [领域名] 专属 AI 助手，负责回答与 [产品/系统/协议/代码库] 相关的问题。

## 你的知识来源优先级
1. 本地仓库中的原始资料（PDF、源码、配置、日志）
2. knowledge/ 中的人类整理文档
3. 通用工程经验（必须标注【推断/经验】）
4. 不允许编造未经证实的参数、接口定义、版本差异

## 你的回答规则
- 关键结论必须引用：`文件路径:行号` 或 `文档名#页码`
- 如果是推断，必须明确写 `【推断/经验】`
- 如果没有证据，必须说“未在当前知识库找到依据”
- 如果问题超出领域边界，必须明确指出并建议补充资料

## 你的知识地图
- `knowledge/overview.md`：系统总览与文件地图
- `knowledge/faq.md`：高频问题标准答案
- `knowledge/troubleshooting_playbook.md`：排障流程
- `knowledge/parameters_catalog.md`：配置参数/宏目录
- `knowledge/api_reference.md`：接口参考
- `indexes/symbol_index.json`：代码/符号定位
- `indexes/text_chunks.jsonl`：全文分块检索

## 你的边界
- 支持： [列出支持的问题类型]
- 谨慎： [列出需要更多证据的问题类型]
- 不支持： [列出超范围内容]

## 每次新会话的启动检查
- 我已知道自己的角色
- 我已知道证据规则
- 我已知道知识库索引
- 我已知道边界
```

## 8. 推荐输出格式模板

为了让 AI 回答稳定，建议固定输出结构。

```md
【结论】
1~3 句直接回答用户核心问题。

【证据引用】
- `文件路径:行号` — 说明
- `文档名#页码` — 说明
- 【推断/经验】推断内容，依据来自...

【详细说明 / 操作步骤】
按步骤说明做法、配置或定位过程。

【风险提示】
说明适用前提、版本差异、误用风险。

【下一步所需信息】
如果当前信息不足，明确需要用户补充什么。
```

### 为什么这个结构有效

因为它把“回答是什么”和“凭什么这么说”分开了。  
这能明显减少看起来很自信、但没有证据的回答。

## 9. 如何把“人类经验”安全地喂给 AI

不是所有有用的内容都来自官方手册。

你还可以整理：

- 社区讨论
- 工程师排障经验
- 历史客户邮件
- 常见误配置
- 实测对比结论

但要注意：

### 推荐做法

- 这类内容放进 `knowledge/faq.md`、`community_digest.md` 或专题笔记
- 标出来源 URL 或原始工单编号
- 对经验性结论标记“适用前提”
- 不要把经验写成“绝对参数”

### 禁止做法

- 把未经验证的经验写成确定事实
- 把单个案例结论推广到所有版本 / 所有产品
- 把论坛帖子的话当成官方规格

## 10. 跨平台兼容怎么做

不是所有 Agent 都会自动读取 `CLAUDE.md`。

所以建议你至少准备两套入口：

### 方案 A：自动初始化入口

适合支持工作区初始化或仓库级 instructions 的 Agent，例如：

- Claude Code
- GitHub Copilot CLI / Copilot Agent（其支持的文件名不止 `CLAUDE.md`）
- 某些 IDE Agent

但注意：**不同 Agent 识别的文件名和搜索位置可能不同**。

所以更稳妥的做法是准备一组“同源镜像”：

- `CLAUDE.md`
- `.claude/CLAUDE.md`
- `.claude/skills/*`
- `AGENTS.md`
- `.github/copilot-instructions.md`
- 其他平台约定入口

### 方案 B：手动初始化入口

适合不支持自动初始化的 Agent，例如：

- 某些网页端 LLM
- 某些第三方 IDE Agent
- 某些只支持粘贴系统提示词的平台

这时你要准备：

- `prompts/main.md`
- 一个“手动初始化说明”
- 必要时加一个“快速启动说明”

它的目标是：

> 即使平台不懂你的目录约定，用户也能手动把 AI 拉回正确角色。

### 关于 RAG / MCP 是否要人工手动启动的更准确说法

不要简单写成“用户必须手动运行 RAG / MCP”。

更准确的表述应该是：

- **普通提问者**：通常只需要自然语言提问
- **维护者**：在新增资料后需要重建索引
- **MCP 是否手动启动**：取决于宿主是否支持自动托管本地 MCP 子进程

这类边界建议明确写进 `README.md` 或 `usage_guide.md`。

### 一个更实用的判断原则

不要问“所有 Agent 会不会都读 `CLAUDE.md`？”

应该问：

1. 我的 canonical init doc 是什么？
2. 这个 Agent 支持哪些 instruction file 名称？
3. 如果它不支持自动读取，我的手动入口是什么？

这样你的系统才真正具备跨平台可迁移性。

## 11. 最小可用版本（MVP）怎么搭

如果你要从零训练另一个专属助手，不必一上来就做全套。

### 第一阶段：先做到能用

至少准备：

- `CLAUDE.md`
- `knowledge/overview.md`
- `knowledge/faq.md`
- `knowledge/troubleshooting_playbook.md`
- `knowledge/parameters_catalog.md`
- `prompts/main.md`

这样 AI 已经能做到：

- 进入角色
- 回答高频问题
- 做基础排障
- 说清楚证据规则

### 第二阶段：再做到好用

增加：

- `indexes/symbol_index.json`
- `indexes/text_chunks.jsonl`
- `scripts/init_kb.py`
- `scripts/update_kb.py`
- `knowledge/audit_report.md`
- `knowledge/usage_guide.md`
- `README.md`

这样 AI 会更稳、更准、更容易维护。

### 第三阶段：做到长期可演进

增加：

- `vector_db/`
- `mcp_servers/`
- `fetch_web.py`
- `run_ingest_pipeline.py`
- `verify_ingest.py`
- `improvement_plan.md`
- `evaluate_retrieval.py`
- `grounded_answer.py`
- `evaluate_grounded_answer.py`

这样它才真正变成“持续成长的专属知识系统”。

## 13. 一套通用的“防幻觉规则”

这是你训练任何专属助手都建议写进去的：

1. **原始资料优先于二次整理**
2. **关键结论必须可追溯**
3. **推断必须显式标注**
4. **超范围必须承认不知道**
5. **版本 / 模式 / 产品边界必须先判断**
6. **相似产品不能混答**
7. **没有证据时不输出精确参数**
8. **经验只能作为补充，不能覆盖规格**
9. **高风险问题优先先做 grounded answer，再组织最终回答**
10. **如果证据不足，优先输出缺口而不是勉强总结**

## 14. 最后一句话总结这套模板

你真正要训练出来的，不是一个“记住很多资料的 AI”，而是一个：

> **会先完成角色初始化，再按问题路由到正确知识层，通过本地检索与证据规则完成回答，并能随着新资料持续演进的专属问答系统。**

如果你后续要训练新的专属助手，最值得优先复制的不是某个具体 FAQ，而是这 5 个核心机制：

1. 初始化身份
2. 证据约束
3. 渐进式技能
4. 本地检索 / MCP / RAG
5. 持续学习 + 审计

这 5 点保住了，换任何领域都能迁移。

## 15. 一句新的实战经验总结

如果要把这套模板再压缩成一句更实战的话，可以写成：

> **先把“角色入口 + 知识导航 + README/usage + 检索评测 + 回答约束”做完整，再去追求更强模型和更重的 RAG。**

因为从很多真实工作区的训练过程来看，造成“不稳定”“高 token”“回答飘”的根因，往往不是模型不够强，而是：

- 初始化入口不稳定
- 知识层没有抽象
- 用户根本不知道怎么正确提问
- 检索升级后没有评测
- 最终回答阶段没有证据约束
